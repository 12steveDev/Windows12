== Ideas para los procesos (tanto "user" como "admin" (que tanto pueden acceder a opcodes especiales)) ==
- `CreateProcess(path, { runAs: "user" | "admin" | "parent", parent: caller_PCB_obj{} })`
|------------------------------|-----------------------|--------------------|
|Proceso Parent (parent.runAs) |CreateProcess( runAs ) |Proceso Hijo Creado |
|------------------------------|-----------------------|--------------------|
| user                         | "user"                | "user"             |
| user                         | "admin"               | Pregunta UAC y luego crea proc "admin" |
| user                         | "parent"              | "user"             |
| admin                        | "user"                | "user"             |
| admin                        | "admin"               | Pregunta UAC (si, siempre) y luego crea proc "admin" |
| admin                        | "parent"              | crea proc "admin" SIN preguntar |
|------------------------------|-----------------------|--------------------|
- Las DLL's no tienen permisos ni nada, solamente usan el runAs del programa que las callea
== Arquitectura de ECSE ==
-> OP Públicos:
- - ADD, CONSOLE_LOG, SET
-> OP Privilegiados (necesitan runAs: "admin"):
- - FS_WRITE_FILE con rutas internas (C:/Windows/, etc...)
-> OP Reservados (necesitan native: true):
- - FS_WRITE_FILE, PM_KILL

- lo "native: true" solamente sirve para que pueda ejecutar los OPCODES reservados directamente, que para un programa normal le tomaría usar funciones exportadas de DLL's nativas
- al cargar las DLL's, ejecutar su código con el PCB de el proceso caller TENIENDO EN CUENTA que si tiene native: true, añadir "native: true" a la copia del pcb del caller que se le pasa a ECSE para que habilite los OPCODEs reservados al ver "pcb.native"

QUIERO OBLIGAR a que los programas usen funciones exportadas por .dll's, pero a la vez que las DLLs si puedan usar los OPCODEs que llaman directo a JS. Ejemplo:
- Notepad quiere guardar archivo
- Notepad NO puede usar "FS_WRITE_FILE" porque no tiene "native: true", por lo que el OPCODE ni existe para el
- Notepad recapacita y hace inteligentemente "CALL WriteFile", a la función WriteFile que le proporcionó kernel32.dll
- ECSE ejecuta el código de la función WriteFile (la cual usa FS_WRITE_FILE)
- ECSE ve que kernel32.dll es native, por lo que ejecuta el OPCODE
- Notepad guardó tu archivo .txt todo qlero con éxito tras varias abstracciones totalmente inútiles pero necesarias, bravo!!
